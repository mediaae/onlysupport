# Doc to Code Workflow

## 1. Intake

Collect every relevant artifact before implementation:

- Patent PDFs, claims, specifications, drawings, office-action material, invention disclosures, DOCX drafts, prior art summaries, tables, datasets, screenshots, and user notes.
- Existing project files, dependency manifests, test commands, build commands, and README instructions.
- Screenshots of enabled MCP servers when provided by the user.

Create an intake summary with:

- File inventory and document type.
- Reading method used: specific MCP/tool name plus capability level, such as `full-structure MCP`, `page-text MCP`, `local parser`, or `fallback`.
- Source quality: selectable text, OCR, scanned pages, missing drawings, damaged tables, or unclear formulae.
- Open questions that block faithful implementation.

## 2. Document Extraction

For DOCX, prefer the strongest available reader. Extract headings, paragraphs, tables, footnotes/endnotes, comments, tracked changes, embedded images, and document properties when relevant. If the available reader cannot expose some of these, record the missing structure and resulting confidence limits.

For PDF, prefer the strongest available reader. Extract text by page, claims, figure references, equations, tables, captions, and OCR warnings. For scanned patents, preserve uncertainty and inspect important pages visually when possible. If the available reader only exposes plain text, record which page-level or figure/table anchors were lost.

Do not rely on a single summary for implementation. Quote or reference exact page, paragraph, claim, figure, table, or heading identifiers in the traceability matrix.

## 3. Requirement Modeling

Separate the document into:

- Core patented mechanism.
- Required inputs and outputs.
- Processing steps and algorithmic constraints.
- Data structures and state transitions.
- UI or presentation requirements.
- Evaluation metrics and proof obligations.
- Non-goals and explicitly excluded behavior.

Flag legal or technical ambiguity early. Ask before choosing between materially different interpretations.

## 4. Conda Environment Construction

Use the user's local Conda installation for the project environment. Derive the environment name from the patent project title, invention name, repository name, or user-provided name.

Name environments with the pattern `patent-<project-slug>` unless the user gives another convention. Keep the slug lowercase, concise, and Conda-safe. Reuse an existing matching environment when available; otherwise ask before creating a new one.

Record:

- Environment name.
- Runtime version.
- Dependency source files.
- Install commands.
- Any fallback if Conda is unavailable.

## 5. Implementation

Read the existing codebase before editing. Preserve architecture, naming style, framework choices, and test patterns unless the patent requires a new component.

Before implementing each technical step, classify its source as `document-explicit`, `user-specified`, or `engineering-glue`. Keep this classification visible in the traceability matrix or implementation notes.

Implement in small, verifiable increments:

- Domain model and data schema.
- Parsing or ingestion layer.
- Patent algorithm or processing pipeline.
- API/service layer.
- GUI-style frontend demonstration interface.
- Proof logging, reproducible outputs, and exportable evidence generated by the code.
- Tests and fixtures.

Keep the code strict but explainable. Prefer deterministic transformations, typed interfaces, explicit validation, and named intermediate values that match the patent terminology.

## 6. Frontend Demonstration Interface

Generate and implement a GUI-form frontend demonstration interface. It must be runnable, domain-specific, and visually distinct. Use `$figma-generate-design` for GUI design after the mechanism and data states are understood.

The frontend must show the implemented patented mechanism through real or reproducible sample data, including inputs, intermediate states, outputs, and test/demo results where applicable.

Technology selection rules:

- If the project already has a frontend, use its existing framework, routing, styling, and build tools.
- If no frontend exists, prefer a lightweight React/Vite frontend for web demonstrations unless the project language or domain clearly calls for another GUI stack.
- If the patent project is desktop, embedded, notebook, scientific, or Python-only, choose the smallest GUI layer that can run reliably while still presenting the mechanism clearly.
- Do not treat a Figma design, screenshot, or static page as the implementation. The frontend must start, render, and connect to real code outputs, deterministic fixtures, or reproducible sample runs.

## 7. Verification

Run the strongest available checks:

- Dependency install only when required and approved.
- Conda environment activation and dependency checks.
- Unit tests for each claim-level behavior.
- Integration tests for document-to-code or data-flow paths.
- GUI smoke tests and screenshots for usability and presentation verification.
- Reproducible sample executions that prove the patent mechanism through code and test data.
- Build, lint, typecheck, and runtime checks.

If a check cannot be run, record why and what risk remains.

## 8. Delivery

Finish with:

- Implemented claim/requirement summary.
- Conda environment name, runtime, and dependency setup notes.
- File changes and generated artifacts.
- Verification commands and results.
- Code/test proof materials, including samples and outputs.
- Required deliverables checklist from `references/deliverables.md`.
- Remaining ambiguities or missing source material.
- Clear note that the result is a technical proof artifact, not legal advice.
